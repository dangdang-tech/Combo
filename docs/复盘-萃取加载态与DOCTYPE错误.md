# 复盘 · 萃取步骤加载态 + 「<!DOCTYPE」错误修复

> 流程:debug(查根因)→ 设计 → 开发 → 测试。

## A. 「Unexpected token '<', <!DOCTYPE」错误
**现象**:网页报这个,像 JSON 解析炸了。
**查证**:服务端所有响应都是 `json()` 或纯文本 `not found`,**从不返回 HTML**。所以 `<!DOCTYPE` 来自 **Railway 边缘**——后端不可达(重启/redeploy 瞬间)时,网关返回 HTML 502 页。我连续 `railway up` 部署多次,用户恰在那 15–30s 窗口请求,就中招。
**实测排除边缘超时**:线上跑真实萃取 **128s 仍返回正常 JSON 200**,边缘不截断长请求 → 确证是「重启瞬时 502」,非超时。
**修复**:`api()` 检查 `content-type`,非 JSON(网关 HTML / 502/503/504)→ 返回可读「服务暂时不可用(可能正在重启)——等几秒点重试」,不再抛 `<!DOCTYPE`。

> 教训:redeploy 会造成短暂 502;前端任何 `fetch().json()` 都得容忍网关 HTML。

## B. 萃取加载态(先设计后开发)
**为什么需要**:萃取是真·慢(12 段就 128s;120 段更久),旧态只有一行「Pi 在反推…」+ 转圈,2 分钟空等像卡死。
**设计**:复用 step1 已确认的「分阶段清单」语言,为「慢 + 思考型」定制:
- 四阶段映射真实管线:**精读会话(S1,有 N/total 真实进度)→ 归纳能力图谱 → 多轮校验(稳定性,3 轮)→ 排序产出**
- 加**「已用 Ns」计时器**(治「卡死了吗」焦虑)
- 实时状态文案随阶段变(精读 47/120 → 归纳出 15 类 → 第 2/3 轮)
- 贴项目气质的安抚行:「只基于真实历史,不编造」
**开发(真实流式,非定时假装)**:
- 后端 `runExtraction` 加 `onProgress` → `extractEmit(id,…)` → SSE `/api/extract/progress`
- 网页 `EventSource` 订阅,驱动 `extractLoader`;POST 仍返回最终 candidates
- 关键:精读阶段的 `N/total` 是真实进度条(`readObservations` 每读完一段回调)

## 测试(全绿)
- SSE 序列实测:`read → progress 1/8…8/8 → taxonomy(15类) → verify → round 1/3,2/3,3/3` ✓
- 浏览器:loader 实时推进(精读✓→归纳⟳,已用 27s),截图匹配设计 A;完成落 1 候选;**零 console 错误** ✓
- 无候选时给「回去多导一些」退路;失败带「重试」
- 单元 25/25;集成 38/38(萃取相关断言受 LLM 非确定性影响偶发,重跑稳定)

## 一句话
慢任务的加载态要给「真实进度 + 已用时 + 它在干嘛」,不是转圈;`<!DOCTYPE` 是网关 502 的伪装,前端必须容忍非 JSON。
