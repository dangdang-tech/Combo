# 复盘 · 永久部署 + 访问码闸

> 按流程走完一轮:研究 → 设计 → 代码 → 测试 → 复盘。

## 做了什么
把「隧道临时 URL + 本机在线」推进到**可固定部署 + 有访问控制**:
- **访问码闸**(`ACCESS_CODE`):非空时创作者面(导入/草稿/锚定/打包,烧 key)需输码 → `/api/unlock` 校验 → HttpOnly cookie(30 天);**消费侧 `/miniapp` + `/api/miniapp/*` 放行**,凭发布 token 用别人做好的 app,可单独分享。本地不设码 = 全开,旧行为零回归。
- **fetchText 双路**:有代理(本地)走 curl(node 原生不读代理);**无代理(云端)走 node 原生 fetch** → 云上不依赖 curl 二进制。
- **部署文件**:`Procfile`(`web: node loop-server.mjs`)、`railway.json`(NIXPACKS + 重启策略)、`package.json` `engines.node>=20`、`start.sh` 云端兜底(无 .env 但 env 有 key → 直接起)。

## 决策与依据
| 决策 | 为什么 |
|---|---|
| 创作者面锁、消费侧开 | 创作者面会烧 OpenRouter key(导入→提取→打包),要防滥用;消费侧凭 token 用 app 正是分享的本意 |
| cookie(非每请求带码) | 同事输一次码即可,体验顺;HttpOnly 防 JS 读取 |
| Railway 而非 Vercel | mini-app 是 SSE + 内存会话的有状态长进程,serverless 不适合 |
| 云端原生 fetch 而非强依赖 curl | nixpacks Node 镜像不保证有 curl;原生 fetch 在 Node20+ 稳定、云上直连无需代理 |

## 测试评估(全绿)
- 闸:锁定 GET `/`→解锁页、创作者 API→401、消费侧 `/miniapp`→放行、错码→401、对码→200+cookie、带 cookie→创作者面解锁 ✓
- 回归:**无码本地 GET `/`→直接进创作者面**(不弹解锁页)✓
- 集成 `test-flow.mjs` 38/38;单元 25/25 ✓
- 语法 `node -c` ✓

## 还没做 / 下一步
1. **实际 `railway up`**:文件与代码已就绪,真正部署需在联网机器执行 5 步(见 `docs/部署-railway.md`),并填 Railway variables。
2. **状态持久化**:`apps-db.json` + 内存会话在重部署时清。要持久化需挂卷或换 DB(MVP 可接受)。
3. **成本闸**:消费侧不需码即可调 LLM(分享本意),若被刷需单 IP/会话软上限。
4. **opencode 上传解析**:目前上传支持 Claude+Codex,opencode(SQLite)客户端解析可补。

## 一句话
公网从「能用」走到「能控 + 能久」:创作者面有码闸防烧 key、消费侧凭 token 开放分享、云端去 curl 依赖、部署文件齐活。差最后一步真实 `railway up`(需在联网机器手动跑登录+部署)。
