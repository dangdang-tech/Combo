# 复盘 · 加载态/异常态全面审查 + 卡死修复

> 用户卡在加载态。要求复查:① 异常态管理 ② 异常态兜底 ③ 整体友好性。

## 根因
1. **redeploy 重置 `apps-db.json`**(无持久卷)→ 用户在流程中时,app 状态被清空。
2. 多个 loader **没有"状态丢失/空结果"出口**,冻结在转圈。最严重:`fsRevisit` 空结果时 `L.stop()` 只停计时器、**卡死无退路**,且**页面加载就自动跑**(没触发也卡)。

## 三问逐项

### ① 异常态管理(改前 vs 改后)
| 环节 | 改前 | 改后 |
|---|---|---|
| fsRevisit(复访读文件夹) | 空结果/异常 → 冻结 | 每条路径都有出口:成功 / `L.fail` 带重试 / 回 step1 |
| step2load(萃取) | app 没了仍轮询 8 分钟 | `failGone`:404 no app → **1 秒**回第一步;被打断 → 自愈重触发 |
| step5load(发布) | `S.manifest` 空 → 抛错卡转圈 | `step5error`:空规范/app 没了 → 重试/回上一步/回第一步 |
| step4(结构化) | **本就是范本** | `structureSlow`(3 退路 + 保留已生成)/`structureError`/2 次重试上限 |

### ② 异常态兜底(现在每个 loader 保证有出口)
- importLoader / extractLoader 的 `fail()` 都插入「重试」按钮。
- "app 没了" 全链路识别(`error==="no app"`)→ 快速回第一步,不傻等。
- 萃取被重启打断 → "not extracted" 自愈重触发一次。
- 轮询总上限 8→6 分钟;连接抖动(SSE 无 data 的 error)正确忽略、不误判失败。

### ③ 友好性
- **去掉"页面一打开就自动弹加载态"**(fsRevisit 改成「↻ 用上次授权的文件夹继续」按钮)——不再惊讶/卡住。
- 失败文案都给**下一步动作**(重试 / 回某步 / 换方式),不留死胡同。
- 计时器 + 实时状态(慢任务)让等待可感知。

## 根因修复:状态持久化
- `DB_PATH` 可配 → 挂 **Railway 持久卷 `/data`**,`apps-db.json` 跨重启/重部署保留。
- 启动清僵尸 `extracting` 标志后 `save()` 落盘;读 db 失败不崩、从空开始。
- **实测铁证**:建 app → `railway redeploy` 重启容器 → app 仍在。

## 验证
- bogus id → **1s** 回第一步带退路(非冻结)✓
- 持久化:重启后 app 存活 ✓;404 no app 正确 ✓
- 单元 25/25;语法/console 干净 ✓

## 一句话
卡死的本质是「状态没了 + loader 不认丢失态」。修法两手:**让状态别没**(持久卷)+ **让每个 loader 都认得'没了/空了'并给退路**(failGone / fail+retry / 回某步)。原则:任何加载态都必须有出口,绝不停在转圈。
