# Raw→能力提取层 · 执行思路:Human-Anchored 协同发现

> 这不是又一个聚类/抽取战术(v1-v4 都是),而是**框架转向**:把目标从「算法自治地发现稳定能力集」换成「算法出便宜草稿 → 创作者一次锚定 → 之后增量分类/提名审核」。
> 前面 v1-v4 不作废,**降级为"草稿引擎"**(门槛骤降),各归其位见 §5。
> 关联:`raw-to-capability-技术方案.md`(v3 草稿引擎)、`crune-原理与实现-深度研究.md`、M3 `distill-to-manifest.mjs`。

---

## 0. 一句话

**便宜粗糙的草稿 → 创作者一次性确认/删改(= 持久锚点)→ 新会话只往锚点里分类、或提名新候选等人审。**
能力集只通过**人审过的显式操作**变化。稳定性、鲁棒性、粒度——**从根上消失**,因为锚是人,不是算法。

---

## 1. 为什么换框架(不是逃避,是对准)

v1→v4 一直在同一个框架里换战术:更好的无监督发现 + 更稳的稳定性度量。退到产品层看,这个框架选错了三处:

1. **目标错位**:产品要的不是"稳定发现全部能力",是**让创作者看到几个候选说"对,这就是我,我要打包这个"**。是 **precision@few + 创作者共鸣**,不是全集召回+稳定性。长尾稳不稳无所谓。
2. **忽略了最高质信号**:"哪件事值得打包成产品去卖"**不在数据里**——是市场/身份判断,只创作者知道。我们却搭精密无监督机器去猜一个人 30 秒能确认的事。
3. **难题是自找的**:稳定性、数据增长鲁棒性——只要**锚点是人不是算法**,这俩直接不存在。

> 魔法不丢:魔法在那份**你没想到的草稿**,人只是**点头**——而点头建立了信任,信任才是他敢拿去卖的前提。一份全自动但他不信的清单,比一份他共签的草稿更糟。
> 与既有原则一致:**agent-friendly、人定方向 AI 出活**;人在环只在**定义产品身份**这步(一次性 onboarding,有动力、不烦),不是执行时反复打扰。

## 2. 架构总览

```
① 导入(已有)
   │  sessionIndex
   ▼
② 草稿引擎(便宜,粗糙可接受 —— 门槛已降)        ← 复用 v3 / crune 骨架,任选
   │  draftCandidates[] {name, intent, evidence[], confidence}
   ▼
③ 锚定交互(核心新部件,一次性,分钟级)          ← 创作者 triage:确认/改名/合并/拆分/删除/新增
   │  人的判断进入这里 —— 全流程最高价值的一步
   ▼
④ 锚点存储(创作者共签的能力集 = 持久 taxonomy)
   │  CapabilityAnchor[] {id, name, intent, slots, evidence, status}
   ▼
⑤ 打包桥(每个确认能力 → M3 manifest + {answer.X} 槽)→ 发布/消费闭环(已有)
   ▲
   │  增量回路:
⑥ 新会话进来 → S1 观察 → 分类进锚点(便宜、稳,因锚固定)→ 增量 evidence
              匹配不上 → 进【提名审核队列】,攒够/定期由人 30 秒审 → ADD 新锚
```

**关键**:②③ 是 onboarding(一次),④⑤⑥ 是日常(增量)。算法只做"草稿"和"分类"——都便宜;人只做"锚定"和"审名"——都低频高价值。

## 3. 各部件

### 3.1 草稿引擎(②)—— bar 已降,可换
- **职责变了**:不再要求"准且稳",只要求"**进大致正确的球场,让人能一眼收口**"。这把 v1-v4 全部困难(稳定性/鲁棒性/粒度)从"必须做对"降为"够用就行"。
- **直接复用**:现有 **v3** 即可(实测 12 稳定 + 8 待定)。在新框架里 **8 个"待定"不再是失败,而是 8 个额外候选**交给人——Jaccard 门从**拦路**降为**给候选排个置信序**的提示。
- 也可换 **crune 确定性骨架**(更便宜,$0 发现层)或最简单的一次 LLM。**草稿引擎可插拔**,因为人在后面兜。
- 产出统一为 `draftCandidates[] = {name, intent, evidence:[sessionRef], confidence: 高|中|低, suggested_slots?}`。

### 3.2 锚定交互(③)—— 核心新部件
创作者在**一屏**内 triage 草稿,几分钟搞定。设计要点:
- **默认收敛**:只默认勾选高置信的少数(~5),低置信的折叠在"可能还有"区——避免 50 个候选淹没人。
- **evidence 一键可验**:每个候选展开就是它来自哪些真实会话(复用我们已做的 `from_segments`/evidence 回链)——让创作者**5 秒确认**而非研究。
- **6 个自然操作**:`确认` / `改名` / `合并两个` / `拆成两个` / `删除(不是我)` / `新增(你漏了)`。
- **零摩擦**:像"我们干了脏活,你只管点头/微调",不是填表。
- 产出 = `CapabilityAnchor[]`,带 `origin: draft|human-added`、创作者的命名为准(身份感)。

### 3.3 锚点存储(④)—— 数据模型
```js
CapabilityAnchor {
  id, name,                      // 创作者确认/改写的名字(身份,锚定后不漂)
  intent,                        // 一句话它干嘛
  slots: ["a","b"],              // 运行时输入槽(打包用)
  evidence: [{path,title,source,date,project}],   // 来自的真实会话(可增量追加)
  status: "confirmed" | "retired",
  origin: "draft" | "human-added",
  manifest_id?,                  // 打包后回填
  createdAt, updatedAt
}
```
**锚点 = 这个创作者的固定 taxonomy。** 它就是 v3"建一次 taxonomy"的位置,但**锚是人审过的,不是 LLM 一次生成的**——质量和信任都更高。

### 3.4 增量分类 + 提名审核队列(⑥)—— 持久价值
- 新会话 → `S1 观察`(已有)→ **分类进锚点**(给定固定锚点,把这段归到 0~多个 id;便宜、稳,因锚固定)→ 命中则**追加 evidence**(能力的"复现度"自然增长,且是真实计数)。
- **匹配不上**(没有锚点能解释这段)→ 进 `ReviewQueue`:`{candidate, evidence, proposed_op: ADD | MERGE_INTO(id) | SPLIT(id), reason}`。
- 攒够 K 个 / 定期 → 给创作者**30 秒审一批**:ADD 新锚 / 并入旧锚 / 忽略。
- **能力集只通过人审的显式操作变化** → 不存在"能力自己解体"(数据增长鲁棒性问题消失)。这是 mem0/ExpeL 的 ADD/UPDATE/NOOP 成长库,但**带人审门**。

### 3.5 打包桥(⑤)—— 接已有闭环
每个 `confirmed` 锚 → `distill-to-manifest` 生成 `AgenticAppManifest`,`slots → {answer.X} → required_context` → 发布/消费(已有)。锚的 `intent + evidence` 还能喂给"写 manifest"那一步当上下文。

## 4. 这一刀解决了什么(逐条 → 非问题)

| 我们四版搏斗的难题 | 在 human-anchored 下 |
|---|---|
| **稳定性**(Jaccard 门、共识 K 轮) | 非问题:人是锚。Jaccard 降为草稿的置信提示。 |
| **数据增长鲁棒性**(crune 致命伤) | 非问题:集合只靠人审的显式操作变,新数据只增 evidence / 提名。 |
| **粒度对不齐**(大桶 vs 一次性) | 人一眼合并/拆分,不用算法猜"可打包的正确粒度"(那本就是产品判断)。 |
| **算法精度/成本** | bar 暴跌:草稿够用就行 → 可用最便宜引擎,不必 $0.066 追完美。 |
| **命名漂移** | 命名一次、由人定,锚定后不再生成。 |

## 5. 旧工作怎么复用(不浪费)

| 既有产出 | 新角色 |
|---|---|
| S1 全量精读(observations) | 留用:喂草稿 + 喂增量分类 |
| v3 锚定 taxonomy 管线 | **降级为草稿引擎**(③ 之前的那一版),门槛降低 |
| evidence / from_segments 回链 | **升级为锚定交互的核验工具**(人 5 秒确认的关键) |
| Jaccard 稳定性度量 | **重定位为草稿候选的置信排序**,不再当门 |
| crune 确定性骨架(研究) | 备选草稿引擎(更便宜),或与 v3 二选一/按源选 |
| distill-to-manifest(M3) | 不变:确认锚 → manifest |
| 多源导入 + 计量 | 不变 |

## 6. 诚实的风险与边界

- **草稿地板**:草稿若是垃圾,人也锚不出好东西。**草稿仍需"好到能被认出"**——不是不要质量,是不要完美。这是真实下限。
- **懒确认**:创作者一键"全确认"→ 垃圾锚。缓解:默认只勾少数高置信、evidence 默认可见、编辑零摩擦、别一次塞 50 个。
- **冷启动**:第一个创作者无锚 → 草稿就是 bootstrap,首次锚定即建集。
- **人在环 ≠ 烦人**:严格限定在 onboarding 锚定 + 低频提名审核两处;执行/消费时绝不打扰(守住 fast-autonomy)。
- **每创作者独立锚**:锚是个人身份,不需要跨用户共享 taxonomy(和 Clio 的跨用户聚合是两回事)→ 单用户场景天然成立。

---

## 7. 里程碑

- **M1 草稿引擎适配**:把 v3 输出统一成 `draftCandidates`(稳定+待定都进,带 confidence)。
- **M2 锚定交互**:一屏 triage(6 操作 + evidence 展开)→ 产出 `CapabilityAnchor[]`,存 `anchors.json`。
- **M3 打包桥**:确认锚 → manifest → 发布闭环跑通。
- **M4 增量分类**:新会话 → 分类进锚 → 追加 evidence;匹配不上进 ReviewQueue。
- **M5 提名审核**:队列攒够 → 一屏 30 秒审 → ADD/MERGE/忽略。
- 验收口径**换了**:不再是 Jaccard≥0.8,而是 **① 创作者锚定耗时(目标<3 分钟)② 确认后能力的"这是我"认同率 ③ 增量分类的人工纠正率**。

---

## 8. 一句话

我们一直在问"怎么让算法更准更稳",真正的杠杆是**换问题**:算法出便宜草稿、创作者一次锚定、之后增量分类。这样 v1-v4 的所有困难从"必须攻克"降为"够用就行",而稳定性/鲁棒性从根上消失——**因为锚是人**。验收也随之从"算法指标"变成"人锚得多快、多认同、改得多少"。
