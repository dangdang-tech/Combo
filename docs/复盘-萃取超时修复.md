# 复盘 · 萃取 240s 超时的稳健修复

> 用户报"前端等待超时(240s)"。查日志 → 多 agent 工作流核验设计 → 实施 → 测试。

## 根因(查 railway 日志)
```
提取 v3 ✓ 243.2s · 149次 · 138594tok · $0.0746 · Jaccard 0.844 PASS · 稳定12/待定8
```
**后端萃取实际成功**(120 段、12 个稳定能力、已落盘),耗时 **243.2s**,但前端 `api()` 的 **240s** AbortController 在后端完成前 **3 秒** abort。纯客户端超时,结果其实已 `save()`。
- 关键事实:Node http 里客户端 abort **不取消**服务端已在跑的 async `runExtraction`(handler 没注册 req close、没透传 signal),所以后端照样跑完保存。
- 时间构成:S1 精读(120 段/并发 8)+ **Taxonomy ~107s** + 分类 3 轮 ~30s。120 段萃取本就 4+ 分钟,任何固定 HTTP 超时都脆。

## 修复(工作流核验后,GO)
1. **缓存命中**:`POST /api/extract` 顶部,已萃取且 `sessionSig`(会话集长度)未变且非 force → 秒回缓存。直接救回"前端超时但后端已完成"。
2. **异步触发 + SSE 交付**:有 sessionIndex 时设 `extracting` 标志、**后台跑、立即返回 `{started:true}`**;完成经 SSE `done` 通知,前端 `GET /api/extract/result` 拉取。**去掉 HTTP 超时天花板**——4 分钟也不超时(SSE 20s ping 保活)。
3. **并发防抖**:`extracting` 时再 POST 返回 `alreadyRunning`,不重复烧 $0.07;启动清僵尸标志;重导入清 extraction 缓存。
4. **前端**:靠 SSE `done`→GET result;`error` 事件 **guard 连接抖动**(仅有 `data` 才算真错,否则是自动重连);15s 轮询兜底 + 8min 总上限;缓存/降级走 POST 直接渲染。降级(粘贴态)保持同步。

## 工作流核验出的竞态(均已处理)
- done 错过 → onopen-先于-POST 订阅 + GET 兜底
- 丢帧 → 结果已落盘,GET 恢复
- 并发 POST → 内存标志拦
- 僵尸标志 → 启动清扫
- 陈旧缓存 → sessionSig 失效
- onerror 碰撞 → 无 data 即忽略

## 测试(全绿)
- POST 0.1s 返回 started;并发 alreadyRunning;SSE done(144s)→GET result→1 候选;**缓存 0.00s 秒回**;force 重跑;浏览器 finish→render 4 候选;零 console;单元 25/25。

## 注意
- **Railway redeploy 会重置 `apps-db.json`**(无持久卷)→ 用户那次超时的旧结果在重部署后已丢;但本修复让**今后**的萃取不再超时(异步+SSE),同容器内断线也靠缓存秒回。状态持久化(挂卷/换 DB)是另一段确定性工作。

## 一句话
慢任务别用固定 HTTP 超时撑——**触发即返回、结果落盘、SSE 通知、GET 拉取、缓存兜底**;客户端 abort 不杀后端,所以结果不丢,只是要给前端一条取回它的路。
