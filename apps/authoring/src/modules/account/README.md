# modules/account — 邮箱登录、账号与会话写入

这个模块是第一方认证的唯一写入方。它请求并验证邮箱六位验证码，在首次验证时创建业务用户和邮箱身份，签发七天固定会话，并提供当前用户与幂等登出接口。本模块不实现密码、手机号、第三方登录、Bearer 凭据或 refresh 流程。

## 文件

- `routes.ts` 只注册四个认证端点。三条 POST 路由使用四 KiB 请求体上限、JSON 守卫和精确浏览器来源守卫，四条路由都在解析请求体前设置 `Cache-Control: no-store`。
- `handlers.ts` 校验共享请求契约，把业务结果映射成统一响应信封和 HTTP 状态。生产环境只设置或清除主机限定的 `__Host-cb_session`，本地 HTTP 开发测试显式使用 `cb_session`；两者都不设置 Domain 并使用根路径。它不记录邮箱、验证码、Cookie 或供应商错误。
- `auth-crypto.ts` 实现邮箱域名的 IDNA 规范化、带版本域标签的 HMAC、六位验证码、随机创作者账号、不透明会话值、会话摘要、常量时间比较和目标 advisory lock 键。
- `service.ts` 编排认证流程。请求验证码时先执行 Redis 客户端软限流和 PostgreSQL 目标硬预算，再在事务外调用邮件适配器，最后按投递结果激活或失效待投递挑战；验证验证码时允许 Redis 故障降级，并把一次消费、用户查建和会话签发交给同一数据库事务。
- `repo.ts` 收拢 `users`、`auth_identities`、`auth_otp_challenges`、`auth_sessions` 与 `auth_audit_events` 的 SQL。它固定使用全站预算锁和目标锁顺序，执行六十秒冷却、小时与 UTC 日预算、五次失败上限、单次消费、随机账号冲突重试、停用检查、会话撤销和低敏审计。投递完成乱序时，旧 pending 挑战看到已经激活的新挑战后只会失效自身，不能覆盖新验证码。

## 上下游

路由由 `bootstrap/routes.ts` 挂到 `/api/v1`。handler 从 `platform/infra/index.ts` 注入的 PostgreSQL、Redis 认证限流器和 Resend HTTP 适配器取得依赖，并使用 `platform/http/` 的来源、JSON、错误信封和路由注册工具。

共享包 `@cb/shared` 提供邮箱请求、验证结果、当前用户、登出结果、Cookie 常量和安全回跳规则。PostgreSQL 是身份、挑战和会话的事实源，并独占验证码请求的目标预算；Redis 保存验证码请求的客户端摘要计数，以及验证码验证的客户端和目标摘要计数；Resend 只负责接收一次邮件投递请求。

## 验证码登录流程

1. 服务端规范化邮箱，并分别计算目标、验证码和客户端地址摘要。
2. 服务端先检查 Redis 客户端窗口，再在 PostgreSQL 中按固定锁序检查目标冷却和预算并写入不可验证的待投递挑战。
3. 服务端在数据库事务外调用 Resend。只有供应商受理时，第二段事务才失效旧活动码并激活新码；永久拒绝返回统一受理结果，系统故障返回 503。
4. 验证事务锁定目标和当前活动挑战。错码原子增加次数并在第五次失效，正确码先消费挑战，再查建用户和邮箱身份。
5. 账号未停用时，事务写入七天会话摘要和成功审计。事务提交后，handler 才把会话原文写入 HttpOnly Cookie。
