## 三、整条链路(数据怎么流)

```
[一段对话/历史]
   │  ① POST /api/import 或 /api/import-history
   ▼
RawInput(原始文本 / 会话标题摘要)         ← 存在 apps[id].raw
   │  ② POST /api/extract   →  Pi 反推
   ▼
candidates[]（多个能力候选:name/tagline/confidence/type/role）
   │  ③ 前端选一个 index + 一段话描述
   │  ④ POST /api/structure  →  Pi 结构化（重 LLM,带 40s 超时兜底）
   ▼
draft（name/tagline/target_users[]/questions[]/output_spec/tags[]/instructions(含 {answer.X}))
   │       └─→ distillToManifest()  →  AgenticAppManifest v0.1
   │  ④ 创作者可编辑 → POST /api/save-draft 重建 manifest
   │  ⑤ POST /api/eval     →  Pi 用示例输入试跑一次(Auto-Eval)
   │  ⑤ POST /api/publish  →  生成 token + 直链
   ▼
published mini-app
   │  ⑤/消费 POST /api/run（消费者填槽 → promptCompiler 拼 prompt → Pi 真跑）
   ▼
artifact（产物 markdown）
```

**关键中转件 = manifest**:`distill-to-manifest.mjs` 把蒸馏结果翻成运行时唯一认的 `AgenticAppManifest v0.1`。指令里的 `{answer.X}` → `interaction.required_context`(消费侧表单字段);带槽指令 → `skill_set[0].steps`(promptCompiler 填槽的模板)。

## 四、Pi / OpenRouter 到底怎么用的(重点澄清)

`pi-exec.mjs` 是**唯一调模型的地方**,两种调用形态,都走 **OpenRouter** provider:

**(A) 单轮补全 `run()` —— 用 `@earendil-works/pi-ai` 的 `complete()`**
用于蒸馏 / 结构化 / 试跑 / 消费 run —— 这些都是"一次问、一次答"。
```js
import { getModel, complete } from "@earendil-works/pi-ai";
const m = getModel("openrouter", "anthropic/claude-3.5-haiku");
const res = await complete(m, { systemPrompt, messages:[{role:"user",content:userInput}] }, { apiKey: OPENROUTER_API_KEY });
```

**(B) 真 agent 基座 `runAgent()` —— 用 `@earendil-works/pi-agent-core` 的 `Agent`(带工具、多步)**
**「能力提取」已升级成真 agent**:不再看标题猜,而是给它两把工具(`list_sessions` / `read_session`),它**自己决定翻哪几段真实会话内容**,读完再萃取 → 不漏东西。
```js
import { Agent } from "@earendil-works/pi-agent-core";
export async function runAgent({ systemPrompt, userInput, tools, model, timeoutMs }) {
  const agent = new Agent({ initialState: { systemPrompt, model: getModel("openrouter", model), tools } });
  let buf = ""; agent.subscribe(ev => { /* 累积 assistant 文本增量 */ });
  return Promise.race([ agent.prompt(userInput).then(() => ({ text: buf })), 超时(timeoutMs) ]);
}
```
- 工具定义在 `loop-server.mjs` 的 `extractTools(idx)`:`{name, description, parameters: Type.Object({...}), execute}`。
- 实测:21s 内 agent 调 `list_sessions` 看全貌 → `read_session` 读 6-10 段真实内容 → 出候选(命中 VC 复盘 16 段、generative UI 研究 21 段等真实主线,远比标题推断准)。
- **韧性**:agentic 解析/超时失败 → 自动降级回 (A) 的标题提取,永不 500;返回里 `agentic:true/false` 标明走了哪条。

> 仍是单轮的步骤(结构化/试跑/消费),将来要让 **mini-app 自己的 agent** 也能调工具,把 (B) 的 `runAgent()` 复用过去即可——基座已就位。

## 五、一键导入(真扫历史,多 agent,不是 mock)

`importHistory()` 串起 **4 个来源**,各自独立 try/catch(某家没装/格式变了不影响其它家),统一成 `{title,count,date,path,source}`:

| 来源 | 位置 | 格式 | 解析 |
|---|---|---|---|
| **claude** | `~/.claude/projects/**/*.jsonl` | 每行事件 | `aiTitle` 当标题,`"role":"user"` 计数 |
| **codex** | `~/.codex/sessions/**/*.jsonl` | rollout `{type,payload}` | `response_item`+`payload.role=user`,跳过注入的 AGENTS.md(`#`/`<` 开头) |
| **opencode** | `~/.local/share/opencode/opencode.db` | **SQLite** | `sqlite3 -json` 读 `session.title` / `message.data` / `part.data` |
| **hermes** | `~/.hermes` 等(探测) | jsonl | 存在才扫,本机暂无 → 0 段 |

- `stats` 多了 `by_source`(各源段数),前端导入页用色标签 + chips 展示来源拆分。
- 导入后 **top 120 段(带 path + source)** 存进 `apps[id].sessionIndex`,提取 agent 用工具按 source 分发读真实内容(`readSessionContent` 按来源走不同解析,见 §四 B / §六)。
- 实测:claude 553 + codex 117 + opencode 2 = **672 段 / 1.7w 条 / 65 项目**。

**仍可深化**:`read_session` 每段只取前 12 条用户消息、各截 400 字;codex 标题用首条真实用户消息前 50 字。

## 六、日志 + 超时(为什么不再"卡很久很久")

**统一日志**(`pi-exec.log`,带时刻+毫秒,写进 `loop.log`):
- 每个 API 进出:`→ POST /api/structure` … `← 200 /api/structure 5.8s`。
- 每次模型调用带 label + 计时:`structure ▶ … 上限40s` → `structure ✓ 5.8s`,超时打 `⏱超时`。
- agentic 提取打**每轮思考 + 每次工具调用**:`🔧 read_session #32 「…」← 1213字`。
- → 任何一步卡在哪、卡多久、读了哪些会话,`tail -f loop.log` 一眼可见。

**两层超时,前后端都不挂死**:
1. **服务端**:`run({timeoutMs})` 用 `Promise.race`(`complete()` 不理 abort,故用 race)。结构化 40s / eval 45s / run 60s / 提取 180s。结构化超时 → 返回"骨架"让创作者手填(Figma `383:139` 恢复态:手填 / 重生成 / 再等等)。
2. **客户端(本次新增 —— 这才是上次"卡很久很久"的真因)**:`api()` 旧版 `await fetch` **没有超时**,后端一旦重启/卡住就无限挂起,前端"已 Ns"永远转。现加 `AbortController`(结构化 70s / 提取 200s / eval·run 90s),超时→返回 `{error}` 落到"重试"态。
   > 实测结构化服务端只 5-6s,根本没碰 40s;之前的多分钟卡死 = 浏览器连了被重启的旧 server,fetch 无超时干等。
